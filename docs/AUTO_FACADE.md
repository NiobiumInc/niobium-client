# Auto-Facade: Transparent and Cooperative Record/Replay for OpenFHE Programs

The auto-facade (`libniobium_client_autofacade.a`, built when
`NIOBIUM_CLIENT_WITH_AUTO_FACADE=ON` — the default, so a plain `make release`
already includes it) removes the manual tagging boilerplate from
record/replay. It has **two modes**:

- **Transparent mode** — zero source changes. The facade intercepts FHE
  operations through the instrumented OpenFHE headers and drives the whole
  `niobium::compiler()` lifecycle itself, configured by a YAML file named in
  the `NIOBIUM_CONFIG` env var (generated by the `tools/nbcc.py` wrapper).
- **Cooperative mode** — one source call, `niobium::compiler().enable_auto_tagging()`.
  The deserialize hooks capture the crypto context and tag keys/inputs
  automatically, but your code keeps ownership of `start()` / `probe()` /
  `stop()` / `replay()` / `result()`. This is the mode the nb DSL generates
  (`dsl_fhe/`), and what the README's auto-tagging paragraph refers to.

**When to use which approach:**

| | Transparent | Cooperative | Manual `niobium::compiler()` SDK |
|---|---|---|---|
| Source code changes | None | `enable_auto_tagging()` after `init()` | Full session API |
| Config mechanism | YAML via `nbcc.py` / `NIOBIUM_CONFIG` | C++ (host owns lifecycle) | C++ API calls |
| Input/key/context tagging | Automatic | Automatic | Manual `tag_input`/`tag_keys`/`capture_crypto_context` |
| Cache params, record/replay gate | Automatic | Manual (`cache_parameters()`, `is_cache_valid()`) | Manual |
| Output capture | Automatic (via `Decrypt` or ciphertext `SerializeToFile`) | Manual `probe()` / `result()` | Manual `probe()` / `result()` |
| Skip expensive plaintext work on replay | Automatic | Manual `is_cache_valid()` check | Manual `is_cache_valid()` check |

Use transparent mode for quick integration with an existing binary you don't
want to touch. Use cooperative mode when generating or writing host code that
should stay in control (the DSL does this). Use the fully manual SDK
(see `examples/bootstrap/`, `examples/mult/`, `examples/simple_ops/`) when you
want explicit control over every tag.

## Where this lives in the code

- **`libnbfhetch`** (in `vendor/niobium-fhetch`, see
  `vendor/niobium-fhetch/src/auto_facade.cpp`) provides *weak* no-op
  implementations of the `niobium_auto::*` hook set, so a program that only
  links libnbfhetch still links cleanly and the hooks stay inert.
- **`libniobium_client_autofacade`** (this repo,
  `src/auto_facade/AutoFacade.cpp`) provides *strong* definitions for those
  hooks. When an executable links both, the strong symbols win and the
  facade becomes active. Because nothing in user code references these
  symbols by name (they are called only from the instrumented headers), the
  static archive must be linked **whole-archive**
  (`--whole-archive` / `-force_load`) or the linker drops it — see
  `examples/CMakeLists.txt` (the `ciphers_ops_server_auto` target) and the
  DSL-generated `dsl_fhe/examples/*/nb_out/CMakeLists.txt`. It also needs
  `yaml-cpp`.
- **Hook activation is compile-time**: the interception points live in
  *header-only* code in the instrumented OpenFHE branch
  (`niobium_auto_hooks.h`, `niobium_auto_scheme.h`, `ciphertext-ser.h`,
  `cryptocontext-ser.h`, installed under
  `vendor/lib/openfhe/include/openfhe/pke/`) and are guarded by
  `#ifdef OPENFHE_CPROBES`. Your translation units must be compiled with
  `-DOPENFHE_CPROBES` or no hook ever fires (see "Linking your own
  application" in the top-level README).

Porting note: this was lifted from
[`niobium-compiler` PR #1035](https://github.com/NiobiumInc/niobium-compiler/pull/1035).
The compiler-specific bits (FPGA/QEMU targets, hardware-mode flags,
`cached_key()`-based key reuse, the multi-path config search) were dropped to
keep the client port small; for those, use the compiler flow.

## Transparent mode

### Quick Start

**1. Build** (auto-facade is on by default):
```bash
make release          # or: make config && make build   (Debug)
```

**2. Run with `nbcc.py`** — generates a config file, sets `NIOBIUM_CONFIG`, and execs the program:
```bash
python3 tools/nbcc.py --name my_program --cache param1=value1 \
    -- ./build/my_program [args...]
```

**3. First run records the FHE operations.** A second run with the same
`nbcc.py` args is a cache hit — the FHE compute is **no-oped** and the result
ciphertexts are reconstructed by the FHETCH simulator replay.

**4. No source changes needed.** See
[`../examples/auto/ciphers_ops_server_auto.cpp`](../examples/auto/ciphers_ops_server_auto.cpp)
for the reference example (zero `niobium::` calls), and
`make test-auto-ciphers-release` for the end-to-end harness.

### Config Discovery

The facade reads exactly one location: the **`NIOBIUM_CONFIG` env var**,
holding the path to a YAML config file. (`nbcc.py` sets it for you.)

**If `NIOBIUM_CONFIG` is unset or the file is missing, the facade stays
dormant** — no compiler init, no recording, no replay. This is deliberate:
client/keygen binaries linked against the facade should not activate it.
There is no CWD / executable-directory config-file search in this port.

### Config File Reference

```yaml
program:
  name:        string   # Program identity — included in cache key. Default: "auto_facade"
  version:     string   # Program version — included in cache key. Default: "1.0"
  description: string   # Human-readable description. Default: "Auto-facade program"

# Arbitrary key-value pairs that distinguish recordings from each other.
# All entries are included in the cache key, so changing any value
# invalidates the cache and triggers a fresh recording.
cache_parameters:
  <key>: <value>        # e.g. wl: "1", op: "ADD"

# Accepted for compatibility, but inert in this port: key tagging happens
# automatically via tag_keys() when the first ciphertext is deserialized.
keys:
  eval_mult:         string
  eval_automorphism: string

# Compiler flags — converted to synthetic argc/argv and passed to
# Compiler::init(). All fields optional; omitted fields use init() defaults.
compiler:
  target:       local          # local = in-process FHETCH simulator (default);
                               # any other value dispatches replay to nbcc_fhetch_replay
  optimization: "3"            # -O level
  registers:    "32"           # --registers
  memory:       "16"           # --memory (GB)
  niobium_hw:   true           # --niobium_hw
  hollow:       false          # enable_hollow_mode (read directly, not passed to init)
  fence:        true           # --fence / --no-fence
  noop:         "0"            # --noop
  multiplier:   "shoup"        # --multiplier (standard | shoup)
  config_sectors: "4"          # --config-sectors
  binary_json:  true           # --binary-json / --ascii-json
  no_cereal_binary: false      # --no-cereal-binary
  transform_bin_to_json: false # --transform-bin-to-json
  no_preserve_input_ciphertexts: false
  formal:       false          # --formal
  lock_timing:  false          # --lock-timing
  no_ring_dim_check: false     # --no-ring-dim-check (needed for toy ring dims)
```

### `nbcc.py` Wrapper

`tools/nbcc.py` generates deterministic config files from CLI flags. Same
flags always produce the same config file (content-hashed), so recording and
replay runs find the same cache automatically.

```bash
python3 tools/nbcc.py --name NAME \
    [--version V] [--description D] \
    [--cache KEY=VALUE ...] \
    [--keys-mult PATH] [--keys-auto PATH] \
    [--target TARGET] [-O LEVEL] \
    [--registers N] [--memory GB] \
    [--niobium-hw] [--hollow] \
    [--fence | --no-fence] [--no-ring-dim-check] \
    [--fhetch-driver PATH [--driver-cc PATH] [--driver-ring-dim N] [--driver-output NAME:PATH ...]] \
    -- EXECUTABLE [ARGS...]
```

(Run `nbcc.py --help` for the remaining pass-through flags: `--noop`,
`--multiplier`, `--config-sectors`, `--binary-json`/`--ascii-json`, etc.)

**Behavior:**
1. Builds a YAML string from the CLI options
2. Writes to `.nbcc/<sha256[:16]>.yml` in CWD
3. Sets `NIOBIUM_CONFIG=.nbcc/<hash>.yml`
4. `os.execvp()` the target executable

If `--fhetch-driver` is given and the recorded trace already exists, `nbcc.py`
execs the `fhetch_driver` helper directly instead of the target binary
(cache-valid replay without re-running the program at all).

**Example** (this is what `make test-auto-ciphers-release` runs):
```bash
# Recording
python3 tools/nbcc.py --name auto_ops_ADD --cache wl=TOY --cache op=ADD \
    --keys-mult io/toy/keys/mk.bin --keys-auto io/toy/keys/rk.bin \
    --no-ring-dim-check \
    -- ./build/examples/ciphers_ops_server_auto 0 output_a.bin output_b.bin 10 ADD 0

# Replay (same command — deterministic config → cache hit)
python3 tools/nbcc.py --name auto_ops_ADD --cache wl=TOY --cache op=ADD \
    --keys-mult io/toy/keys/mk.bin --keys-auto io/toy/keys/rk.bin \
    --no-ring-dim-check \
    -- ./build/examples/ciphers_ops_server_auto 0 output_a.bin output_b.bin 10 ADD 0
```

### How It Works

`lazy_init()` runs once (`std::call_once`), triggered by the first
CryptoContext deserialization:

1. Reads `NIOBIUM_CONFIG`; no config → **dormant** (or **cooperative** if
   `enable_auto_tagging()` was called — see below), return immediately
2. Parses YAML and builds synthetic `argc`/`argv` from the `compiler:` section
3. Calls `Compiler::init(argc, argv)`, then `set_program_info()`,
   `set_build_info()`, `cache_parameters()`, `capture_crypto_context()`
4. Checks `is_cache_valid()` to determine mode **before** setting hollow:
   - **Miss (record)** → sets `enable_hollow_mode()` if `hollow: true`,
     `start()` fires after the CC hook completes
   - **Hit (replay)** → sets `g_replay_mode = true`; the actual `replay()`
     call (argument-less — the client replays through the local FHETCH
     simulator, or `nbcc_fhetch_replay` for non-`local` targets) is
     **deferred** until the first output serialize/decrypt needs a result
5. The `NiobiumAutoScheme` proxy is installed **only in replay mode**; the
   recording path uses the real scheme, and the trace comes from the
   `OPENFHE_CPROBES` probes
6. Key tagging (`tag_keys()`) is anchored to the **first ciphertext
   deserialize** — by then the program has loaded its eval keys into the
   context — so record and replay runs allocate identical FHETCH addresses

### Replay No-Ops

When `g_replay_mode = true`, all FHE compute operations go through
`NiobiumAutoScheme` (and its sub-proxies `NiobiumAutoFHE`,
`NiobiumAutoAdvancedSHE`), which return a lightweight dummy ciphertext
(preserving CC pointer, key tag, and encoding type) without executing any
polynomial arithmetic. The `g_replay_noop_count` atomic counter tracks how
many operations were no-oped.

### Output Capture (Probe and Result)

Output ciphertexts are captured at two interception points — **whichever the
program hits first** for a given ciphertext:

**Ciphertext `SerializeToFile`** (hook: `on_serialize_ciphertext`) — when the
program writes a result ciphertext to disk:
- **Recording:** calls `compiler().probe(filename_stem, ct)` and proceeds with the file write
- **Replay:** triggers the deferred replay, calls
  `compiler().result(cc, filename_stem, hw_ct)` and writes the reconstructed
  result to the file

**`Decrypt`** (hook: `on_decrypt`) — when the program decrypts a result:
- **Recording:** calls `compiler().probe(name, ct)` (if not already probed via
  serialize), then proceeds with normal Decrypt (paused out of the trace)
- **Replay:** calls `compiler().result(cc, name, hw_ct)`, substitutes the
  dummy with the reconstructed result, then runs the real Decrypt on it

A ciphertext is only probed once — if `SerializeToFile` already probed it,
`Decrypt` reuses the same probe name. Probe names from `SerializeToFile` are
filename stems; probe names from `Decrypt` are auto-incremented (`output_0`,
`output_1`, ...); substituted ciphertexts are remembered so repeat decrypts
(including the instrumented `Decrypt`'s own re-entrant call on the substituted
result) keep their recorded name instead of minting a new one.

Plaintexts created via `MakePackedPlaintext` / `MakeCKKSPackedPlaintext` are
also intercepted (`on_make_plaintext`): during recording they are tagged as
inputs with deterministic `pt_<N>` names (the encode itself is paused out of
the trace); during replay these calls return a null plaintext and the recorded
values are rehydrated from disk.

### CryptoContext Normalization

Each ciphertext file serialized independently by cereal contains its own
CryptoContext instance. To prevent OpenFHE's `TypeCheck` from rejecting
operations between ciphertexts from different files,
`on_deserialize_ciphertext` normalizes all ciphertext CC pointers to the
shared `g_cc`. (The facade's own I/O — e.g. loading
`serialized_probes/<name>.ct` during `result()` — is excluded from input
tagging by a re-entrancy guard.)

### atexit Handler

In transparent mode `AutoFacade.cpp` registers a process-exit handler that
calls `compiler().stop()` to finalize and persist the recording. Before
stopping, it swaps the `NiobiumAutoScheme` proxy back to the real scheme so
cereal serialization succeeds. (Cooperative mode registers no atexit handler —
the host calls `stop()` itself.)

### Recording and Replay Flows

#### Recording

```
Program starts
    │
    ├─ DeserializeFromFile(cc.bin, cc)
    │       └─ on_deserialize_crypto_context(cc)
    │               └─ lazy_init() → loads config, init(), determines RECORD mode
    │               └─ start() → recording begins (real scheme, probe-driven)
    │
    ├─ DeserializeFromFile(input.bin, ct)
    │       └─ on_deserialize_ciphertext() → normalizes CC,
    │          tag_keys() on the first ct, tag_input() per ct
    │
    ├─ EvalAdd(ct1, ct2)
    │       └─ real math; OPENFHE_CPROBES probes record the polynomial ops
    │
    ├─ Decrypt(sk, result, &pt)
    │       └─ on_decrypt() → probe("output_0", result)
    │          pause recording → real Decrypt → resume recording
    │
    └─ Program exits
            └─ atexit → stop() → recording finalized
```

#### Replay

```
Program starts
    │
    ├─ DeserializeFromFile(cc.bin, cc)
    │       └─ lazy_init() → is_cache_valid() → true → REPLAY mode
    │          installs NiobiumAutoScheme proxy (replay is deferred)
    │
    ├─ DeserializeFromFile(input.bin, ct)
    │       └─ normalizes CC, refreshes changed input files
    │
    ├─ EvalAdd(ct1, ct2)
    │       └─ NiobiumAutoScheme → dummy() → no-op (noop_count++)
    │
    ├─ Decrypt(sk, dummy_result, &pt)
    │       └─ on_decrypt() → first output access → replay() runs the trace
    │          → result(cc, "output_0", hw_ct) → real Decrypt on hw_ct
    │
    └─ Program exits normally
```

## Cooperative mode (auto-tagging for host-driven code)

Calling `niobium::compiler().enable_auto_tagging()` right after `init()` puts
the facade in **cooperative** mode: no YAML config is read, no scheme proxy is
installed, and no atexit handler is registered. The deserialize hooks do the
tagging; your code does everything else:

- `capture_crypto_context()` fires automatically when the CryptoContext is
  deserialized;
- `tag_keys()` fires automatically on the **first ciphertext deserialize**
  (after your `DeserializeEvalMultKey` / `DeserializeEvalAutomorphismKey`
  calls have populated the context);
- each deserialized ciphertext is tagged as an input, named from its file
  path (separators replaced with `_`, `.bin` stripped) plus a per-path
  `_load_<N>` counter; plaintexts from `MakePlaintext` are tagged `pt_<N>`;
- your code still calls `start()` (after the input loads), `probe()`,
  `stop()`, `is_cache_valid()`, `replay()`, and `result()` — the same
  lifecycle as the README's minimal example, minus the manual
  `capture_crypto_context` / `tag_input` / `tag_keys` calls.

This is exactly what the nb DSL's generated `@hardware` server stages do —
see any `dsl_fhe/examples/*/nb_out/compute.cpp` for the canonical sequence and
the accompanying generated `CMakeLists.txt` for the build wiring
(`OPENFHE_CPROBES` define + whole-archive `libniobium_client_autofacade` +
`yaml-cpp`).

On a cache-valid cooperative run, `replay()` uses the in-process FHETCH
simulator by default; if the `NBCC_FHETCH_DRIVER` env var names a
`fhetch_driver` binary (built with niobium-fhetch), the replay is spawned
through that helper instead (the DSL Makefile sets this).

## Build Instructions

The auto-facade is part of the default build:

```bash
make release          # Release
make config && make build   # Debug
```

The CMake switch is `NIOBIUM_CLIENT_WITH_AUTO_FACADE` (default `ON`); turning
it `OFF` skips `src/auto_facade/` and the `ciphers_ops_*` auto example. After
changing instrumented-OpenFHE headers, do a clean rebuild (`make clean` first).

## Testing

```bash
# keygen → encrypt → record → cache-hit replay, zero niobium:: in user code
make test-auto-ciphers-release

# other ops / operands
make test-auto-ciphers-release AUTO_OP=MUL AUTO_EXPECTED=21
make test-auto-ciphers-release AUTO_OP=ADDI AUTO_A=9 AUTO_IMM=2 AUTO_EXPECTED=11
```

The target (part of `make test-release` via `test-client-release`) drives
`ciphers_ops_cache_keys` → `ciphers_ops_client` → two `nbcc.py`-wrapped runs
of `ciphers_ops_server_auto` (record, then deterministic cache-hit replay) and
verifies the decrypted result both times.

## Limitations

- **One recording per process.** `lazy_init` fires once. Programs that process
  multiple independent datasets in a single run will have them folded into one
  recording.
- **Expensive plaintext work still runs during recording.**
  `MakePackedPlaintext`/`Decrypt` are only excluded from the instruction
  *trace*, not skipped entirely. To skip the full computation on replay, check
  `compiler().is_cache_valid()` yourself (cooperative / manual SDK pattern).
- **Config comes from `NIOBIUM_CONFIG` only.** There is no config-file search
  in CWD or next to the executable.
